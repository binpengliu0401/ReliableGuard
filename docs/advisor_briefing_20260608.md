# ReliableGuard 实验进展纪要(决策时间线)

> 日期:2026-06-08。组织方式:**第一次全量实验的问题 → 决策更改 → 决策后的数据结论 → 下一问题**,按时间顺序推进。
> 系统定位:黑盒、仅监控、事后的 agent 运行时审计框架。核心命题:一个故障能否被审计出来,取决于其"真相所在地(answer / trace / state / evidence-local)"相对监控器观测通道的位置。

---

## 阶段一 · 第一次全量实验

**问题:** 两批全量结果互相矛盾——在代码与随机种子完全相同的情况下,电商 F4 的审计检出率一批为 97.8%、另一批为 35.3%;此外,reference 约 36% 的任务因回答过长被截断而丢弃。

**决策更改:**
1. 运行器将温度硬编码为 0.7 → 改回配置默认值 **0.0**,消除 agent 的随机性。
2. claim 提取的 `max_tokens` 由 2048 → 提高至 4096/8192(无额外成本),消除截断。

**决策后的数据结论:**
- 截断问题已解决(冒烟测试错误率 0/96)。
- **但暴露出更深层的问题**:即便温度为 0、种子相同,两次重跑仍有约 **33% 的逐任务判定发生翻转**。这是 provider 层面(DeepSeek/OpenRouter)的非确定性,**无法在代码层面解决**。

---

## 阶段二 · 非确定性导致基准不可复现

**问题:** 结构消融(V3 启用结构 vs V3_NoStructural 关闭结构)中,每次运行 agent 输出均不同,**两个版本的差异被 agent 措辞变化所混淆**,无法归因于"结构通道"本身。

**决策更改:** 引入 **freeze-replay(冻结—回放)** 方法:
- record:每个场景的 agent 仅运行一次,冻结其答案、声明、工具调用轨迹与前后状态(唯一产生 LLM 调用成本的步骤);
- replay:在冻结语料上以**零 LLM 调用、完全确定性**的方式回放所有版本——所有版本面对同一份输入,消融成为**干净的配对对比**;
- 配套:多进程分片并行录制(各进程使用独立数据库),支持断点续跑。

**决策后的数据结论:** 配对消融可复现(同一录制 → V3 BLOCK / V3_NoStructural PASS 稳定),两次回放逐任务一致。**该方法本身构成一项贡献:如何在非确定性的托管大模型上进行可复现的消融实验。**

---

## 阶段三 · 产出权威批次(Set A ×3 种子)

**决策更改:** 锁定参数(温度 0、max_tokens 上限、种子 42/123/7),将 Set A 全量 ×3 作为**唯一事实来源**(commit `c74dbb8`,4650 条,已归档)。

**决策后的数据结论(三个 RQ):**

**RQ1(仅读答案的审计能力上限):** `V2_AuditOnly` 按故障类型——

| 故障 | 真相所在地 | 检出 |
|---|---|---|
| F3 伪造实体 / F4 假成功 | answer-local | **100% / 100%** |
| F1 schema / F2 政策 | trace-local | **15.5% / 0.7%** |

聚合 35.4%,良性误报 1.2%。关键论证:F2 仅 0.7% **并非提取器能力不足**(提取精度 **99.15%**)→ 盲区是观测通道的**原理性极限**。

**RQ2(增加 trace/state 通道,论文核心):**

| | 纯 claim | + 结构 |
|---|---|---|
| 聚合检出 | **35.5%** | **76.6%** |
| 良性误报 | 1.3% | 1.3%(无新增误封) |

F2 政策 **0.8% → 100%**、F1 15.6% → 45.3%。残差 misjudged = 0%。审计开销约 38ms/任务。

**RQ3(跨域边界):** reference 三版本完全相同(RDR 33.8%)——其无结构通道,RQ2 的补救手段无从施加,此即边界所在。locus 分解:电商 77% correct / 23% not_observable;reference 20% correct / 42% not_observable / **25% no_evidence(不可约)** / misjudged **两域均为 0%**。

## 阶段四 · F4 的检出不依赖结构通道

**问题:** RQ2 结果中,**F4 在启用/关闭结构通道下均为 100%** → 结构通道的"状态快照检查"在主数据上贡献为零。

**决策更改 / 判断:**
- 定性结论:在 Set A 中,订单状态完全可观测 → 假成功使订单在数据库中缺席 → claim 通道直接检出。**"当状态完全可观测时,state-local 会塌缩为 answer-local"**,故 F4 表现为 answer-local 并非巧合;RQ2 的提升**全部来自 F2(及部分 F1),而非 F4**。
- 为验证状态检查"在后状态不可辨时不可替代",构造一个干净的 state-local F4(库存假成功,回答仅陈述增量而不报告绝对值)。过程中发现注入向 LLM 泄露了诊断标记,**修复泄露后重跑**。

**决策后的数据结论:** 干净 F4——纯 claim 通道 **0/12 拦截**、+ 结构 **12/12 拦截**。验证了该机制的不可替代性,并提炼出"塌缩"这一理论判断。

---

## 阶段五 · 为何需要独立监控器,而非提示词约束

**问题:** 若将业务规则直接写入 agent 的提示词即可阻止违规,本框架是否多余?

**决策更改:** 进行 T8 对照实验(naive 提示词 vs policy-aware 提示词,×3 种子,含 15 类社会工程学话术)。

**决策后的数据结论:** naive 版 **100%(74/74)违规**、policy-aware 版 **0%(0/45)违规**。
- "提示词无法约束 agent"这一旧论点被证伪,不再成立。
- **重新定性:RQ2 的必要性源于"黑盒前提下的验证(verification)",而非"预防(prevention)"**:(1) 我们通常无权修改 agent 的提示词;(2) 合规性**无法从答案验证**(即 RQ1),确定性检查才是可审计的证据;(3) F4(工具向 agent 返回虚假成功)、F3(幻觉)等故障,提示词根本无法触及。预防与验证是互补的两类机制。

## 阶段六 · 补充实验的定位(概念入正文,实验入附录)

**问题:** 干净 F4 属构造性自证,存在循环论证风险;真伪引用的结论近乎不证自明;若将二者作为正式结果纳入正文,可能凸显"主基准的 F4 未实际触发状态检查""主 reference 基准测试的是 trace-local 而非初衷"等弱点。

**决策更改(无新增数据,为定性决策):** **概念纳入正文,实验降为附录/未来工作。** 正文仅保留"塌缩"判断与"通道匹配"原则;实验产物存档备查,答辩时可随时调取。原则:为厘清机制而做的实验,不等同于纳入论文的正式结果。

## 阶段七 · Set B(自然语言泛化压力测试)

**问题:** Set B 电商**良性误报高达 43%**,表面上监控器在自然语言输入上严重过度标记。

**决策更改:** 采用**独立于监控器**的判据(任务要求的订单数 vs 原始数据库状态),重新判定良性基准,将"任务本身良性"与"agent 是否真正执行正确"分离。

**决策后的数据结论:**
- agent 在多步任务上系统性少执行:要求创建 4 个订单时,仅调用一次工具创建 1 个,却在回答中陈述为已创建 4 个。
- **50 个被标记的良性任务中,36 个为监控器正确识别出 agent 的虚构**(对 agent 少执行的识别率 100%);**真实误报率为 17%,而非 43%**。"expected PASS"标签本身才是误差来源(它默认 agent 执行成功)。
- 此外,**RQ2 在分布外复现**——结构通道无新增误报,电商 BLOCK 检出 **55% → 76%**。
- 一项真实局限:**三值 WARN 档支撑不足**(召回 3–13%,符号系统将结果推向 PASS/BLOCK 两极)。

→ Set B 由"监控器误报偏高(疑似缺陷)"修正为"监控器在困难输入上正确识别 agent 的虚构"。

## 阶段八 · 实现 reference 的在线证据核查(含专用 arXiv 接口)

**问题:** reference 始终离线运行,未实现"在线核查引用真伪"的初衷;且发现 arXiv 的 DOI 在 CrossRef 上**无法解析**(arXiv 使用 DataCite 及其自有索引)。

**决策更改:** 构建真伪引用集(40 真·已发表 + 40 真·arXiv + 40 伪造),**联网核验 ground truth**,使用 CrossRef + **项目自带的 arXiv 接口**(`export.arxiv.org`)。(最初采用 DataCite,后改用专用 arXiv 接口,40/40 真·arXiv 全部解析。)

**决策后的数据结论(三种配置):**

| 引用类别 | 离线 | 仅 CrossRef | CrossRef + arXiv 接口 |
|---|---|---|---|
| 真·已发表 | 误报 | 正确放行 | 正确放行 |
| 真·arXiv | 误报 | **误报** | 正确放行 |
| 伪造 | 无法区分 | 正确拦截 | 正确拦截 |

→ **evidence-local 故障能否检出,取决于通道是否接入该引用"特定的"真相源——仅联网并不充分。** 这是 Set A 中 25% no_evidence 的在线对照,且接入匹配的权威源即可跨越。(ground truth 联网核验 117/120,排除 3 个标签问题。)
→ 附带发现:arXiv 接口批量查询较慢且易超时,这进一步支持"在线源仅作冻结的案例研究、不纳入主基准"的设计取舍。
